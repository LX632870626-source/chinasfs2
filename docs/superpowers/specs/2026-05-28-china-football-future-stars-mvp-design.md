# 中国足球未来之星抖音小程序 MVP 设计

## 背景

项目目标是做一个面向球迷和家长的“中国足球未来之星”抖音小程序。第一版聚焦内容浏览和运营维护，让用户可以发现小球员、追踪青少年赛事，也让管理员可以持续维护资料和审核外部提交。

当前工作目录还不是小程序工程，已有内容主要是一份中国 U23 球员海外效力报告生成脚本和输出文件。因此第一阶段先定义产品范围、信息架构、数据模型和实现边界。

## 第一版目标

第一版目标是可运营 MVP，而不是纯演示原型。

需要支持：

- 前台浏览未来之星和青少年比赛日历。
- 首页做推荐流和内容导流，用户可以点击查看更多。
- 管理员通过轻量 Web 后台维护球员、赛事、赛程和提交审核。
- 家长、教练或球迷可以提交球员或赛事线索，提交后进入待审核状态。
- 前台只展示低敏公开信息，后台保留更完整资料。

第一版暂不做：

- 全球青训营资料板块。
- 用户社区、评论、打分、排行榜。
- 球员认领和完整用户体系。
- 视频搬运、热门赛事回放聚合。
- 复杂球员数据统计。

## 用户与定位

主要用户是球迷和家长。产品气质采用热血青训媒体感：首页有内容冲击力，球员故事有关注点，赛程信息清晰可靠，适合抖音小程序里的轻量浏览和转发。

后台用户是管理员，负责录入、编辑、上下架和审核资料。第一版后台只做单一管理员权限，不做多角色。

## 技术路线

推荐路线是：

- 抖音小程序前台。
- 独立轻量 Web 管理后台。
- 后端 API 和数据库。

前台按小程序接口边界设计，只读取公开字段。后台通过管理接口维护完整资料。这样第一版可以形成真实运营闭环，也方便后续接入抖音小程序上线流程。

## 前台信息架构

底部导航建议 3 个 Tab：

1. 首页
2. 未来之星
3. 赛程

资料提交入口不单独作为 Tab，放在首页右上角、首页模块或轻量入口中，避免抢占主浏览流程。

## 首页

首页承担发现和导流作用，不承载全部内容。

首页模块：

- 顶部：应用名“中国足球未来之星”和搜索入口。
- 主视觉：本周重点球员或重点赛事大卡。
- 本周未来之星：横向球员卡片，提供“查看更多”进入完整球员库。
- 近期亮点：展示球员故事、近期表现或编辑推荐摘要。
- 即将开赛：展示未来几场青少年赛事，提供“查看更多”进入完整赛程。
- 资料提交入口：鼓励家长、教练或球迷提交球员和赛事线索。

不设置热门赛事和回放模块。赛程页可以保留官方链接，但不做站内视频搬运和回放聚合，降低资料收集压力和版权风险。

## 未来之星

未来之星 Tab 是完整球员库。

核心能力：

- 搜索球员。
- 按年龄段、位置、球队、地区、技术特点筛选。
- 查看全部公开球员资料。
- 从首页推荐卡片进入球员详情。

球员详情页重点讲清楚：

- 他是谁。
- 踢什么位置。
- 技术特点是什么。
- 为什么值得关注。
- 近期参加过哪些比赛。
- 是否有官方、授权或公开视频链接。

球员详情不堆砌敏感信息，不展示联系方式、详细生日、学校或详细住址。

## 赛程

赛程 Tab 是完整青少年比赛日历。

核心能力：

- 查看近期赛事和比赛。
- 按日期、地区、年龄组、赛事状态筛选。
- 查看赛事详情和赛程详情。
- 在赛事或比赛详情中展示时间、地点、年龄组、参赛队、状态和官方链接。
- 在球员详情中关联重要参赛记录。

第一版不做复杂比分数据库和球员技术统计，重点是“最近有什么可看”和“这个球员参加过哪些重要比赛”。

## 资料提交

资料提交用于收集球员和赛事线索。

提交内容：

- 提交类型：球员或赛事。
- 基本资料。
- 提交人联系方式。
- 说明文字。
- 外部链接或附件线索。

提交后进入待审核状态，不直接公开。管理员审核后，可以标记为已采纳、已拒绝或待处理。审核通过后也不自动发布，而是由管理员手动创建或更新正式资料。

## 后台管理

后台是轻量 Web 管理端，第一版包含 4 个模块。

### 球员管理

- 新增、编辑、上下架球员。
- 设置首页推荐和推荐排序。
- 维护公开资料和后台私有资料。
- 上传或设置封面图。
- 添加技术特点标签。
- 关联比赛记录和视频链接。

### 赛事管理

- 新增、编辑、上下架赛事。
- 维护赛事名称、时间、地点、地区、年龄组、简介、封面图和官方链接。
- 管理赛事下的赛程。

### 赛程管理

- 为赛事添加具体比赛。
- 维护参赛队、比赛时间、比分、状态、直播或官方链接。
- 关联重点球员。

### 提交审核

- 查看用户提交的球员和赛事线索。
- 标记待处理、已采纳、已拒绝。
- 填写管理员备注。
- 根据采纳内容手动创建或更新正式资料。

## 数据模型

### Player 球员

公开字段：

- 姓名或昵称。
- 年龄段。
- 位置。
- 所属球队。
- 地区。
- 技术特点标签。
- 简介。
- 封面图。
- 官方、授权或公开视频链接。
- 是否首页推荐。
- 推荐排序。
- 上架状态。

后台私有字段：

- 真实生日。
- 身高体重。
- 惯用脚。
- 学校或训练机构。
- 联系人。
- 备注。
- 资料来源。
- 公开级别。

### Team 球队或机构

- 名称。
- 地区。
- 类型：俱乐部梯队、校园球队、青训机构等。
- 简介。
- 上架状态。

### Event 赛事

- 赛事名称。
- 年龄组。
- 地区。
- 地点。
- 开始时间。
- 结束时间。
- 状态。
- 简介。
- 封面图。
- 官方链接。
- 上架状态。

### Match 比赛或赛程

- 所属赛事。
- 比赛时间。
- 参赛队伍。
- 比分。
- 状态。
- 官方链接。
- 关联重点球员。

### Submission 资料提交

- 提交类型：球员或赛事。
- 提交人联系方式。
- 提交内容。
- 附件或外部链接。
- 审核状态。
- 管理员备注。
- 创建时间。

### Admin 管理员

- 登录账号。
- 密码哈希。
- 状态。
- 最后登录时间。

## 隐私与内容安全

因为核心对象包括未成年人，第一版采用分级展示。

- 前台只展示低敏公开字段。
- 后台可保存更完整资料，但不能默认公开。
- 联系方式、详细生日、学校、详细住址不在前台展示。
- 视频只保存外部链接，不搬运，不聚合回放，不展示未经授权内容。
- 前台 API 必须避免返回后台私有字段。
- 提交内容必须经过管理员审核。

## API 边界

前台公开接口：

- 获取首页推荐内容。
- 获取球员列表和筛选结果。
- 获取球员详情公开字段。
- 获取赛事和赛程列表。
- 获取赛事或比赛详情公开字段。
- 提交球员或赛事线索。

后台管理接口：

- 管理员登录。
- 球员增删改查和上下架。
- 赛事增删改查和上下架。
- 赛程增删改查。
- 推荐排序管理。
- 提交审核管理。

前台接口和后台接口必须分开，前台不依赖后台私有字段。

## 测试重点

- 前台接口不能返回敏感字段。
- 球员上下架后，前台列表和详情表现正确。
- 首页推荐排序准确。
- 搜索和筛选结果可靠。
- 赛事和赛程按时间排序正确。
- 提交表单必填校验有效。
- 审核状态流转准确。
- 后台登录和未登录访问控制有效。

## 里程碑

1. 建立项目基础结构、数据库模型和后端 API。
2. 完成前台首页、未来之星列表、球员详情、赛程列表和赛事详情。
3. 完成资料提交。
4. 完成后台登录、球员管理、赛事管理、赛程管理和提交审核。
5. 加入基础测试和隐私字段校验。
6. 用示例数据完成一轮端到端验收。

## 未决但不阻塞的问题

- 抖音小程序最终上线时的账号、审核、云服务和域名配置。
- 图片和附件的正式存储方案。
- 是否后续增加全球青训营资料板块。
- 是否后续开放用户登录、球员认领或投稿进度查询。
